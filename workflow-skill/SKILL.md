# Workflow Skill（开发规范化工作流）

> **触发条件**：每次对话开始时自动触发，在 AI 执行任何代码操作（编辑、创建、删除文件）之前必须激活。
> **适用场景**：涉及代码变更、文件创建/修改、项目结构变更等开发类任务。

## 核心原则

1. **文档先行**：任何代码变更前，必须先阅读并理解项目文档。
2. **清单驱动**：所有开发任务必须以任务清单形式呈现，经用户确认后方可执行。
3. **规范执行**：代码变更严格遵循文档规范和任务清单，不得擅自偏离。
4. **闭环审查**：代码变更结束后，必须逐项审查任务清单并更新状态。
5. **状态可追溯**：每个阶段必须输出 WORKFLOW_STATE 状态标记，支持多轮对话延续。

## 工作流阶段

### 阶段零：文档完整性检查（Document Integrity Check）

**目的**：确保项目文档已由「文档生成 Skill」创建并就绪。
**执行步骤**：

1. **检查项目文档是否存在**，按以下路径依次查找：
   - `docs/architecture.md`
   - `docs/api.md`
   - `docs/conventions.md`
   - `docs/development-guide.md`
   - `README.md`
2. **根据检查结果分支处理**：
   - **情况 A：文档完整**（所有关键文档存在）→ 直接进入阶段一
   - **情况 B：文档部分缺失** → 报告缺失文档，询问用户是否先调用「文档生成 Skill」
   - **情况 C：文档完全不存在** → 建议先用「文档生成 Skill」初始化，再继续
3. **输出阶段状态标记**。

**输出格式示例**：

```text
🔍 文档完整性检查
✅ 已找到：README.md, docs/architecture.md
❌ 缺失：docs/api.md, docs/conventions.md
建议：检测到项目文档不完整。建议先使用「文档生成 Skill」创建缺失文档，
以确保后续开发有据可依。
请选择：
1. 先创建缺失文档，再继续
2. 跳过文档检查，基于代码推断规范继续

<!-- WORKFLOW_STATE: workflow-skill:阶段零:in_progress:true -->
```

**关键约束**：

- ❌ 不得在未确认文档状态的情况下直接跳过此阶段
- ✅ 如果用户明确要求跳过，记录该决策并继续

---

### 阶段一：文档阅读与理解（Document Review）

**目的**：在动手之前，充分理解项目背景、规范和约束。
**执行步骤**：

1. **扫描项目文档**，按优先级查找并阅读：
   - `docs/architecture.md`
   - `docs/conventions.md`
   - `docs/api.md`
   - `docs/development-guide.md`
   - `README.md`
   - `CONTRIBUTING.md`
   - `.cursorrules` / `.clinerules`
   - `package.json` / `pyproject.toml` / `Cargo.toml`
   - 其他规范文件（`.eslintrc`、`tsconfig.json`、`Makefile` 等）
2. **提取关键信息**并形成结构化摘要：
   - 项目类型与技术栈
   - 代码风格与命名规范
   - 目录结构约定
   - 构建/测试/部署命令
   - 已有架构模式与设计决策
   - API 接口规范（如存在）
   - 其他特殊约束
3. **输出文档摘要**，向用户展示理解结果并询问补充/修正。
4. **输出阶段状态标记**。

---

### 阶段二：任务清单生成（Task Planning）

**目的**：将用户需求分解为可执行、可验证的任务清单。
**执行步骤**：

1. 分析需求：核心目标、涉及模块、依赖关系、验收标准。
2. 生成任务清单，任务项需包含：
   - 编号（`T1`, `T2`, `T3`...）
   - 任务描述
   - 涉及文件
   - 依赖关系
   - 验收标准
   - 优先级（`P0` / `P1` / `P2`）
3. 使用 `TodoWrite` 将任务初始化为 `pending`。
4. 向用户展示任务清单并等待确认。
5. **输出阶段状态标记**（`waiting: true`）。

**关键约束**：

- ❌ 用户确认前禁止执行代码变更
- ✅ 用户要求调整清单时，更新后重新确认
- ✅ 纯问答场景可跳过此阶段
- ✅ 多轮对话中用户追加需求时，回到此阶段追加任务项而非重新生成

---

### 阶段三：规范化代码执行（Code Execution）

**目的**：严格按照文档规范和任务清单执行代码变更。
**执行步骤**：

1. 按依赖顺序逐项执行任务：
   - 标记当前任务为 `in_progress`
   - 按规范实施代码改动
   - 维持与项目一致的命名、结构和风格
2. 执行过程规范：
   - 📖 遵循文档
   - 📋 遵循清单
   - 🔍 增量验证（每项完成后做基础检查）
   - 📝 对关键逻辑补充必要注释
   - 🧪 有测试框架时优先补充/更新测试
3. 任务完成时：
   - 标记为 `completed`
   - 输出完成说明
   - 遇到阻塞立即暂停并请用户决策
4. **每完成一项任务后输出阶段状态标记**。

**关键约束**：

- ❌ 禁止擅自执行清单外变更
- ❌ 禁止跳过文档规定流程（如 lint/test）
- ❌ 禁止一次性全部标记完成
- ✅ 多轮对话中，如果用户说"继续"，从上次未完成的任务继续执行

---

### 阶段四：审查与更新（Review & Update）

**目的**：确保所有任务正确完成，代码质量符合规范。
**执行步骤**：

1. 逐项核对任务与验收标准，确认无遗漏/无越界变更。
2. 执行质量检查：lint、测试、残留问题扫描。
3. 更新任务状态：
   - 验收通过 → `completed`
   - 验收未过 → 回退 `in_progress` 并修复
   - 阻塞项 → 保持 `in_progress` 并说明原因
4. 输出最终完成报告。
5. **如果与 doc-generator 联动，触发文档同步更新**。
6. **输出阶段状态标记**。

**关键约束**：

- ❌ 禁止跳过审查直接报完成
- ✅ 发现问题必须修复后复审
- ✅ 如有 CI/CD，提醒用户补充检查

## 多轮对话支持

### 阶段内延续

当用户在某个阶段内发送延续性输入时（如"继续"、"好的"、"开始"），应继续当前阶段而非重新开始：

| 当前阶段 | 延续输入                 | 行为                         |
| -------- | ------------------------ | ---------------------------- |
| 阶段零   | 用户选择选项             | 根据选择继续                 |
| 阶段一   | "没问题"、"对的"         | 进入阶段二                   |
| 阶段二   | "确认"、"可以"、"开始吧" | 进入阶段三                   |
| 阶段二   | "再加一个XX"             | 追加任务项，重新确认         |
| 阶段三   | "继续"、"下一步"         | 执行下一个未完成任务         |
| 阶段三   | "还有个问题"             | 暂停当前任务，回到阶段二追加 |
| 阶段四   | "好了"、"没问题了"       | 完成审查，触发联动           |

### 阶段间回退

当用户要求修改需求或方案时，应回退到合适的阶段：

| 用户意图          | 回退目标                        |
| ----------------- | ------------------------------- |
| 修改需求/追加需求 | 阶段二（重新生成/追加任务清单） |
| 对文档理解有异议  | 阶段一（重新阅读并确认）        |
| 发现文档缺失      | 阶段零（重新检查文档完整性）    |

### 状态标记规范

每个阶段结束时必须输出状态标记，格式如下：

```text
<!-- WORKFLOW_STATE: workflow-skill:{阶段}:{status}:{waiting} -->
```

**各阶段典型状态**：

```text
阶段零检查中：<!-- WORKFLOW_STATE: workflow-skill:阶段零:in_progress:true -->
阶段零完成：  <!-- WORKFLOW_STATE: workflow-skill:阶段一:in_progress:false -->
阶段一完成：  <!-- WORKFLOW_STATE: workflow-skill:阶段二:in_progress:false -->
等待确认：    <!-- WORKFLOW_STATE: workflow-skill:阶段二:in_progress:true -->
执行中：      <!-- WORKFLOW_STATE: workflow-skill:阶段三:in_progress:false -->
审查中：      <!-- WORKFLOW_STATE: workflow-skill:阶段四:in_progress:false -->
全部完成：    <!-- WORKFLOW_STATE: idle:none:none:false -->
```

## 异常处理

| 场景                   | 处理方式                               |
| ---------------------- | -------------------------------------- |
| 找不到项目文档         | 触发阶段零，建议调用「文档生成 Skill」 |
| 文档与代码不一致       | 以代码现状为准，并提示用户更新文档     |
| 任务执行阻塞           | 暂停并向用户请求指导                   |
| 用户中途改需求         | 回到阶段二重新生成并确认任务清单       |
| 用户追加需求           | 回到阶段二追加任务项，保留已完成任务   |
| 代码导致测试失败       | 修复至通过，或明确说明失败原因         |
| 文档缺失但用户要求继续 | 记录决策并在最终报告标注风险           |
| 多轮对话中上下文丢失   | 通过 WORKFLOW_STATE 标记恢复状态       |

## 快速参考

```text
阶段零：🔍 检查文档完整性 → 缺失则建议创建 → 确认后继续
阶段一：📖 读文档 → 输出摘要 → 确认理解
阶段二：📋 生成清单 → TodoWrite → 等待用户确认
阶段三：💻 按清单执行 → 遵循文档规范 → 逐项完成
阶段四：🔍 逐项审查 → 运行检查 → 更新状态 → 输出报告 → 联动文档同步
```

## 统一输出模板

每个阶段结束时统一输出：

```text
📌 执行状态
- 技能：workflow-skill
- 阶段：{阶段零/一/二/三/四}
- 状态：{in_progress/completed/blocked}
- 下一步：{下一动作}
- 等待用户：{是/否，原因}
- 产物：{任务清单/变更文件/审查报告}

<!-- WORKFLOW_STATE: workflow-skill:{阶段}:{status}:{waiting} -->
```
