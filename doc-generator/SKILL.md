# Doc Generator Skill（文档生成与维护）

> **触发条件**：
>
> - 被用户手动调用（`@doc-generator`）
> - 被 Workflow Skill 阶段零检测到文档缺失时建议调用
> - Workflow Skill 阶段四（审查更新）结束后自动触发文档同步
>
> **适用场景**：项目初始化、文档缺失补充、代码变更后的文档同步更新。

## 核心原则

1. **文档即代码**：项目文档与代码同等重要，必须保持同步。
2. **增量更新**：代码变更后仅更新受影响的文档章节，不做全量重写。
3. **需求可追溯**：每次需求变更必须记录，形成完整的需求迭代链。
4. **与 Workflow Skill 联动**：文档结构满足 Workflow Skill 阶段零的检查要求。
5. **并发安全**：团队协作时必须遵循并发安全机制，避免文档冲突。

## 文档体系结构

本 Skill 管理的项目文档体系如下：

```text
docs/
├── architecture.md          # 架构设计文档
├── conventions.md           # 编码规范文档
├── api.md                   # API 接口文档
├── development-guide.md     # 开发指南
└── requirements/
    ├── requirements.md      # 需求总览与迭代记录
    └── changelog.md         # 需求变更日志（按时间倒序）
```

> **说明**：`README.md` 不在本 Skill 管理范围内（通常由项目作者手动维护）。

## 并发安全机制

### 更新前检查（Pre-update Check）

每次执行阶段二（文档同步更新）前，**必须**执行以下检查：

1. **Git 状态检查**：
   - 执行 `git status` 检查目标文档是否有未提交的变更
   - 执行 `git pull` 获取远程最新版本
   - 如果存在冲突，暂停更新并向用户报告

2. **文档时间戳比对**：
   - 读取文档头部的「最后更新」时间戳
   - 如果该时间戳晚于本技能开始执行的时间，说明文档已被他人修改
   - 暂停更新，向用户展示差异，询问处理策略

3. **文件指纹校验**：
   - 在首次读取文档时记录文件的行数和关键段落摘要作为"指纹"
   - 在写入前重新读取文件，比对指纹是否一致
   - 如果指纹不一致，说明文件已被他人修改，触发冲突处理流程

### 安全更新策略

根据冲突级别采用不同策略：

| 冲突级别 | 场景                            | 策略                                             |
| -------- | ------------------------------- | ------------------------------------------------ |
| 无冲突   | 修改不同文档                    | 直接更新                                         |
| 低冲突   | 修改同一文档的不同章节          | 逐章节更新，每章节更新前重新读取确认上下文未变   |
| 高冲突   | 修改同一文档的同一章节          | 暂停，展示双方变更，让用户手动合并或选择保留哪方 |
| 需求冲突 | 同时追加 changelog/requirements | 遵循原子追加协议（见下方）                       |

### 冲突处理流程

```text
检测到冲突
  │
  ├─ 低冲突（不同章节）
  │    └─ 重新读取最新文件内容
  │    └─ 基于最新内容定位目标章节
  │    └─ 执行增量更新
  │    └─ 验证写入成功
  │
  ├─ 高冲突（同一章节）
  │    └─ 暂停更新
  │    └─ 向用户展示：
  │         - 当前文件内容（最新版）
  │         - 本技能计划写入的内容
  │         - 冲突的具体位置
  │    └─ 等待用户选择：
  │         1. 以我的变更为准（覆盖）
  │         2. 以已有变更为准（放弃本次更新）
  │         3. 手动合并（用户提供合并后的内容）
  │
  └─ 需求冲突（追加类）
       └─ 遵循原子追加协议
```

### 原子追加协议（用于 changelog.md / requirements.md）

追加类更新必须遵循以下步骤，确保多人同时追加不会丢失数据：

```text
1. 读取文件完整内容，记录末尾 N 行作为"锚点"
2. 在内存中拼接新条目
3. 再次读取文件，比对末尾锚点是否与步骤1一致
   ├─ 一致 → 执行写入
   └─ 不一致 → 用最新内容重新拼接新条目 → 再次验证 → 写入
4. 写入后立即读取验证内容正确性
5. 如果连续 3 次验证失败，暂停并向用户报告冲突
```

### 文档锁定建议（团队层面）

在团队协作场景中，建议采用以下策略减少冲突：

1. **文档更新时机**：建议在个人功能分支完成后、提交 PR 前触发文档同步，而非在开发过程中频繁触发
2. **文档负责人制度**：建议每个迭代指定一名文档负责人，统一执行 doc-generator 的同步更新
3. **分支策略**：文档更新应跟随功能分支，通过 PR 合并而非直接在主分支更新
4. **CI 集成**：建议在 CI 中加入文档一致性检查，在 PR 合并时自动触发文档同步
5. **更新频率控制**：对同一文档的更新间隔建议不少于 5 分钟，避免高频并发

## 工作流阶段

### 阶段一：项目分析与文档初始化（Initialization）

**目的**：分析项目并创建标准文档集合。
**触发条件**：

- 首次使用本 Skill
- Workflow 阶段零判定文档完全缺失

**必须创建的文件**：

- `docs/architecture.md`
- `docs/conventions.md`
- `docs/api.md`
- `docs/development-guide.md`
- `docs/requirements/requirements.md`
- `docs/requirements/changelog.md`

**执行步骤**：

1. 扫描目录与关键配置，识别技术栈、模块、命令。
2. 基于现有代码生成上述 6 份文档。
3. 每份文档写入 `最后更新` 与 `更新原因`。
4. 输出初始化报告（已创建/待补充项）。

### 阶段二：文档同步更新（Sync Update）

**目的**：在代码变更结束后，同步更新受影响的文档。
**触发条件**：

- Workflow Skill 阶段四（审查更新）完成后自动触发
- 用户手动调用 `@doc-generator update`

**执行步骤**：

1. **并发安全检查**（新增，必须执行）：
   - 执行 Git 状态检查
   - 执行文档时间戳比对
   - 执行文件指纹校验
   - 如果检测到冲突，按冲突处理流程处理

2. **获取变更信息**：
   - 从 Workflow Skill 的完成报告中获取：变更的文件列表、任务清单、需求描述
   - 如果没有 Workflow 报告，通过 `git diff` 或文件对比分析变更范围

3. **影响分析**，判断哪些文档需要更新：

| 变更类型                | 需要更新的文档                     |
| ----------------------- | ---------------------------------- |
| 新增/修改模块或架构调整 | `architecture.md`                  |
| 修改命名规范、代码风格  | `conventions.md`                   |
| 新增/修改/删除 API 接口 | `api.md`                           |
| 修改构建命令、开发流程  | `development-guide.md`             |
| 任何需求变更            | `requirements.md` + `changelog.md` |

4. **执行增量更新**：
   - ❌ **禁止全量重写文档**，仅更新受影响的章节
   - 使用精确定位方式更新对应段落
   - 更新文档头部的「最后更新」日期和「更新原因」字段
   - 保持文档的整体结构和格式不变
   - 每个章节更新前重新读取确认上下文未变（低冲突防护）

5. **更新需求记录**（遵循原子追加协议）：
   - 在 `requirements.md` 中新增或更新需求条目
   - 在 `changelog.md` 中追加变更记录
   - 关联 Workflow Skill 的任务编号（如 `T1, T2`）

6. **输出更新报告**：

```text
📄 文档同步更新完成
| 文档 | 更新内容 | 变更类型 |
|------|---------|---------|
| architecture.md | 新增模块 X 的描述 | 章节追加 |
| api.md | 更新 /api/users 接口参数 | 内容修改 |
| requirements.md | 新增需求 v1.3 | 条目新增 |
| changelog.md | 追加 2026-04-01 变更记录 | 条目新增 |
📊 更新统计：
- 更新文档：4 份
- 新增章节：2 个
- 修改段落：3 处
- 需求版本：v1.3
- 冲突检测：{无 / 已自动解决 / 已暂停等待用户}
```

**关键约束**：

- ❌ 禁止删除文档中的现有内容（除非对应的代码已被删除）
- ❌ 禁止修改文档的模板结构（标题层级、表格格式等）
- ❌ 禁止跳过并发安全检查直接更新
- ✅ 如果变更涉及架构性调整，建议用户审阅后再确认更新
- ✅ 如果无法确定是否需要更新某文档，向用户确认

### 阶段三：需求记录（Requirement Tracking）

**目的**：记录和管理需求迭代，形成完整的需求链。
**触发条件**：

- 用户提出新需求时
- Workflow Skill 阶段二生成任务清单时（同步记录需求）
- 代码变更导致需求变更时

**执行步骤**：

1. 在 `requirements.md` 记录版本、需求摘要、优先级、关联任务。
2. 在 `changelog.md` 追加变更条目（时间倒序），遵循原子追加协议。
3. 根据任务执行结果更新子需求状态（`pending` / `done` / `deprecated`）。

**版本号规则**：

- 主版本（v2.0）：重大架构变更或不兼容修改
- 次版本（v1.3）：新增功能或较大需求
- 补丁版本（v1.3.1）：小修改或文档更新

## 与 Workflow Skill 的联动协议

### Workflow Skill → Doc Generator 的调用时机

```text
Workflow 阶段零：文档缺失
  └→ 建议：「检测到文档缺失，建议先使用 @doc-generator 初始化文档」
  └→ 用户确认后，暂停 Workflow，调用 Doc Generator 阶段一
Workflow 阶段四：审查完成
  └→ 自动触发 Doc Generator 阶段二（文档同步更新）
  └→ 同时触发 Doc Generator 阶段三（需求记录）
Workflow 阶段二：任务清单确认
  └→ 同步触发 Doc Generator 阶段三（记录新需求）
```

### Doc Generator → Workflow Skill 的返回

```text
Doc Generator 阶段一完成
  └→ 返回：「文档已就绪，可以继续 Workflow 阶段一（文档阅读）」
Doc Generator 阶段二完成
  └→ 返回：「文档已同步更新，无需额外操作」
```

## 异常处理

| 场景                   | 处理方式                               |
| ---------------------- | -------------------------------------- |
| 项目为空（无代码）     | 生成最小化文档模板，标注「待填充」     |
| 无法判断技术栈         | 向用户确认，基于文件扩展名推断         |
| 文档与代码严重不一致   | 生成差异报告，建议用户确认以哪方为准   |
| 需求描述模糊           | 向用户追问，在需求记录中标注「待澄清」 |
| 文档文件被手动删除     | 重新生成对应文档，在 changelog 中记录  |
| 并发更新冲突（无冲突） | 直接更新                               |
| 并发更新冲突（低冲突） | 重新读取后更新                         |
| 并发更新冲突（高冲突） | 暂停，展示差异，等待用户决策           |
| 并发更新冲突（追加类） | 遵循原子追加协议                       |
| 原子追加连续失败 3 次  | 暂停并向用户报告，建议手动处理         |

## 快速参考

```text
阶段一：🔍 分析项目 → 生成 6 份文档 → 输出初始化报告
阶段二：🔒 并发安全检查 → 获取变更 → 影响分析 → 增量更新文档 → 输出更新报告
阶段三：📝 记录需求 → 原子追加 changelog → 同步需求状态
联动：Workflow 阶段零缺失 → 初始化 → Workflow 阶段四完成 → 并发安全检查 → 同步更新
```
